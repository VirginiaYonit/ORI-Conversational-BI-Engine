# ?? ORI – LLM Intervention Policy**Version**: 0.1**Status**: Governance Draft---## 1. PurposeThis policy defines the conditions under which the Large Language Model (LLM) is allowed to operate within the ORI architecture.The LLM is a **narrative and interpretative component**, not a computational one.---## 2. Separation PrincipleORI enforces a strict architectural separation:* **Deterministic Layer**  Numerical computation, metrics, correlations, divergence* **LLM Layer**  Explanation, synthesis, insight generationThe LLM:* has no direct access to the dataframe* does not perform calculations* does not generate numerical outputs---## 3. Allowed LLM InterventionsThe LLM is activated only when all of the following conditions are met:1. Contract Validator returns `status = OK`2. Deterministic computation has been completed3. No new or undefined KPI is requiredAllowed tasks:* Interpret deterministic results* Generate insights* Provide managerial synthesis (“so what”)* Suggest follow-up analyses or mini-reports* Propose future KPIs (without computing them)---## 4. Forbidden OperationsThe LLM must not:* Generate numerical values* Compute metrics* Use KPIs outside the active focus* Override the contract decision* Modify the analytical context implicitlyAny attempt to produce unsupported numerical content is flagged by the audit layer.---## 5. Human-in-the-LoopKPI selection is always a human decision.The system may:* propose KPI rankings* highlight risks (missing data, unstable definitions)The system does not modify the KPI focus autonomously.---## 6. Audit and MonitoringAll LLM outputs are subject to validation mechanisms:* violation detection* unsupported number detection* out-of-scope KPI detectionThe LLM is not considered a source of numerical truth.---## 7. LLM Blocking ConditionsThe LLM is not activated when:* `status = ASK`* `status = REFUSE`* deterministic computation is not available* the request is out of scope---## 8. Residual RiskPrimary risk:* generation of text that appears complete despite underlying limitationsMitigation:* architectural separation* audit layer* decision logging---## Design NoteORI treats governance as a runtime property, not as documentation.
