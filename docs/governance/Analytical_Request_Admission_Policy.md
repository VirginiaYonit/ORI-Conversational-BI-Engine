# ?? ORI – Analytical Request Admission Policy

**Version**: 0.1
**Status**: Internal Governance Draft

---

## 1. Purpose

This policy defines how ORI evaluates whether an analytical request can be:

* **Accepted** (`OK`)
* **Clarified** (`ASK`)
* **Rejected** (`REFUSE`)

The goal is to ensure consistency, traceability, and control of system behavior.

---

## 2. Core Principle

ORI executes deterministic computations only when **minimum computability conditions** are satisfied within the active analytical context.

If these conditions are not met, the system stops before execution.

---

## 3. Acceptance Criteria (`OK`)

A request is accepted when:

1. At least one KPI is explicitly identified
2. The KPI belongs to the active `kpi_focus`
3. The KPI is computable (`kpi_data` available)
4. For multi-KPI operations (e.g. correlation, divergence):

   * at least 2 computable KPIs are available
5. The requested mode (analytic / tabular) is compatible

---

## 4. Clarification Cases (`ASK`)

ORI requests clarification when:

* no KPI is identified
* the KPI is outside the active focus
* the KPI is not computable
* the operation requires =2 KPIs but fewer are available

In these cases:

* no computation is executed
* no numerical output is produced
* a contract step is logged

---

## 5. Rejection Cases (`REFUSE`)

ORI rejects the request when:

* the requested operation violates system rules
* tabular mode requires unsupported derived metrics
* the request is explicitly disallowed

Rejection is final unless the request or context is modified.

---

## 6. Partial Admission

If a request includes both valid and non-computable KPIs:

* ORI proceeds with computable KPIs only
* excluded KPIs are explicitly declared
* the computation scope is logged

---

## 7. Traceability

Each decision is logged as:

```text
event = "contract_step"
```

Including:

* reason
* involved KPIs
* active mode
* outcome (`OK / ASK / REFUSE`)

---

## 8. Deterministic vs LLM Separation

* numerical computation is strictly deterministic
* the LLM operates only after computation
* no new metrics can be generated by the LLM
* all LLM outputs are subject to audit

---

## Design Note

The contract is not a validation layer.
It is the decision boundary of the system.
